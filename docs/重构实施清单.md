# DataPilot 重构实施清单

## 1. 文档目的

本文档基于《架构优化与 RAG / Milvus 设计方案》，将下一阶段工作拆成可执行的实施清单。

目标不是再做一份架构说明，而是明确：

- 先做什么
- 后做什么
- 每一步产出什么
- 涉及哪些模块
- 完成的判断标准是什么

本文档默认项目当前目标是：

- 先把 DataPilot 打磨成稳定、专业的“生成型数据 Agent 后端”
- 再逐步演进到可执行的“执行型数据 Agent”

## 2. 实施总原则

### 2.1 原则

- 先收口架构边界，再增强能力
- 先治理数据与检索，再优化 Prompt 与模型
- 先抽象接口，再接执行器和调度器
- 先保证线上只读稳定，再考虑自动化初始化

### 2.2 不建议的推进方式

- 一边保留旧结构，一边继续往旧结构上堆功能
- 未拆清 collection 设计前就继续强化 SQL Prompt
- 未补清模板结构前就继续强化任务生成逻辑
- 在线服务继续自动建 collection 和补 schema

## 3. 阶段划分

建议分五个阶段推进：

1. 基础收口与安全卫生
2. RAG / Milvus 专业化重构
3. SQL 与任务生成链路重构
4. 执行扩展口与平台化接口预留
5. 质量、观测与后续演进

## 4. 第一阶段：基础收口与安全卫生

### 4.1 目标

- 让项目边界清晰
- 移除明显安全风险
- 为后续 collection 重构和服务重构建立基础

### 4.2 任务清单

#### T1. 清理配置模板与敏感信息

目标：

- `.env.example` 不再包含疑似真实凭据
- 默认配置体现“本地开发示例”而非“实际运行值”

涉及文件：

- `.env.example`
- `config.py`
- `README.md`

输出物：

- 安全的 `.env.example`
- 更清晰的配置说明

验收标准：

- 仓库中不再出现真实格式凭据
- README 明确说明哪些配置必须由环境提供

#### T2. 明确项目边界与运行模式

目标：

- 统一团队共识：当前阶段是“生成型 Agent”

涉及文件：

- `README.md`
- `docs/architecture-rag-milvus-design.md`

输出物：

- 更新后的 README 定位说明

验收标准：

- 文档不再暗示系统已经具备完整执行闭环

#### T3. 增加基础健康检查定义

目标：

- 明确在线服务需要检查哪些依赖

涉及模块：

- `main.py`
- 新增 `app/infrastructure/health/`

输出物：

- 健康检查设计
- 后续实现所需 DTO

验收标准：

- 已定义 Milvus、Redis、LLM、collection schema 的检查项

### 4.3 第一阶段完成标志

- 仓库安全卫生达标
- 文档定位统一
- 健康检查项定义完成

## 5. 第二阶段：RAG / Milvus 专业化重构

这是当前最重要的阶段。

### 5.1 目标

- 在线服务不再自动建 collection
- collection 按表、字段、模板拆开
- 引入版本化和 alias 机制
- 检索服务从“直接调 Milvus”升级为“仓储抽象 + 检索策略”

### 5.2 任务清单

#### T4. 移除运行时自动建 collection 行为

目标：

- `MilvusService` 启动时只连接和校验，不创建 schema

涉及文件：

- `app/services/milvus_service.py`

输出物：

- 精简后的 Milvus 连接层
- 启动校验逻辑

验收标准：

- 线上代码中不再出现“缺 collection 就创建”的逻辑
- collection 不存在时服务明确报错

#### T5. 设计并落地 collection bootstrap / migration 脚本

目标：

- schema 创建和索引创建由离线脚本负责

建议新增：

- `scripts/bootstrap_milvus.py`
- `scripts/migrate_milvus.py`
- `scripts/validate_milvus_schema.py`

输出物：

- collection 创建脚本
- alias 切换脚本
- schema 校验脚本

验收标准：

- 可独立完成 `dp_table_catalog_v1`
- 可独立完成 `dp_column_catalog_v1`
- 可独立完成 `dp_task_template_v1`
- 可独立完成 alias 切换

#### T6. 重构元数据导入链路

目标：

- 将当前导入脚本改造成按新 schema 入库

涉及文件：

- `scripts/insert_to_milvus.py`
- `scripts/genter_metadata.py`
- 新增离线构建模块

输出物：

- 表级资产构建逻辑
- 字段级资产构建逻辑
- 模板资产构建逻辑

验收标准：

- 不再使用旧的 `metadata_table` / `metadata_column`
- 新导入脚本与在线检索 schema 完全一致

#### T7. 抽象检索仓储层

目标：

- 将在线服务与 Milvus 客户端解耦

建议新增模块：

- `app/infrastructure/milvus/repositories/table_catalog_repository.py`
- `app/infrastructure/milvus/repositories/column_catalog_repository.py`
- `app/infrastructure/milvus/repositories/task_template_repository.py`

建议接口：

- `search_tables`
- `search_columns`
- `search_templates`
- `get_table_by_id`
- `get_columns_by_table_id`

验收标准：

- Node 层不再直接依赖 Milvus search 细节
- 检索过滤、召回、重排可在仓储层集中管理

#### T8. 重构 RagService

目标：

- `RagService` 从“直接 encode + hybrid_search”升级成“查询理解后的组合检索服务”

涉及文件：

- `app/services/rag_service.py`

输出物：

- 表级检索方法
- 字段级检索方法
- 模板检索方法
- 组合上下文构造方法

验收标准：

- SQL 场景支持表级与字段级双通道召回
- 任务场景支持模板 TopK 检索和重排

### 5.3 第二阶段完成标志

- collection 体系升级完成
- 在线服务只读化
- 检索仓储层建立完成
- 新 RAG 数据准备与在线检索链路打通

## 6. 第三阶段：SQL 与任务生成链路重构

### 6.1 目标

- 让 SQL 生成和任务生成都建立在专业的结构化上下文上
- 降低“Prompt 硬扛”的程度

### 6.2 SQL 链路任务

#### T9. 增加 SQL 查询理解层

目标：

- 在生成 SQL 之前先抽取结构化查询意图

建议新增模块：

- `app/domain/sql/query_understanding.py`

输出物：

- 查询意图 DTO
- 时间、指标、维度、过滤、排序、TopN 抽取逻辑

验收标准：

- SQL 生成节点不再直接拿原始用户输入做唯一依据

#### T10. 重构 SQL Node

目标：

- SQL Node 改成“理解 -> 检索 -> 上下文组装 -> 生成 -> 校验”

涉及文件：

- `app/agents/nodes/sql_node.py`

输出物：

- 新的上下文组装逻辑
- 更专业的 Prompt 输入结构

验收标准：

- 能显式区分表候选、字段候选、join hints、时间列等上下文

#### T11. 增强 SQL 安全与校验层

目标：

- 从简单关键词拦截升级为更明确的校验链

涉及文件：

- `app/utils/sql_safety.py`
- 新增 `app/domain/sql/validators/`

验收标准：

- 至少支持：
  - 单语句限制
  - 只读限制
  - 方言判断
  - 黑名单操作拦截
  - 可选白名单 schema / table 控制

### 6.3 任务链路任务

#### T12. 增加任务理解层

目标：

- 在模板检索前先抽取任务类型、源、目标、对象、周期等结构化意图

建议新增模块：

- `app/domain/task/task_understanding.py`

验收标准：

- 任务生成不再完全依赖自由文本检索模板

#### T13. 重构 Task Node

目标：

- Task Node 改成“理解 -> 检索 -> 兼容性过滤 -> 槽位填充 -> 缺参追问 -> JSON 渲染”

涉及文件：

- `app/agents/nodes/task_node.py`

验收标准：

- 不再默认永远只用 Top1 模板
- 缺参追问基于 `required_slots` 和 `slot_schema`

#### T14. 明确 active_task 生命周期

目标：

- 解决当前任务上下文残留问题

涉及文件：

- `app/agents/state.py`
- `app/agents/nodes/task_node.py`
- `main.py`

输出物：

- `collecting`
- `ready`
- `submitted`
- `cancelled`
- `expired`

验收标准：

- 任务完成、取消、切换时上下文规则明确

### 6.4 第三阶段完成标志

- SQL 生成建立在结构化上下文上
- 任务生成建立在结构化模板体系上
- 多轮任务上下文管理规则稳定

## 7. 第四阶段：执行扩展口与平台化接口预留

### 7.1 目标

- 不马上强推执行闭环
- 但要先把接口层留好

### 7.2 任务清单

#### T15. 抽象 SQL 执行器接口

#### T17. 增加审计与解释输出模型

建议新增：

验收标准：

### 7.3 第四阶段完成标志

- 执行器适配口清晰
- 分发器适配口清晰
- 审计与解释模型可扩展

## 8. 第五阶段：质量、观测与后续演进

### 8.2 任务清单

#### T18. 重构健康检查

- 健康检查到位
- 日志与错误分类清晰
- 核心链路有测试保护

## 10. 里程碑定义

### M1：在线只读化

判断标准：

- 在线服务不再创建 collection
- collection schema 与 alias 已固定
- 离线导入链路可用

### M2：专业检索化

判断标准：

- 表、字段、模板三类 collection 生效
- SQL 支持双通道召回
- 任务支持模板 TopK 检索与重排

### M3：专业生成化

判断标准：

- SQL 生成依赖结构化查询理解
- 任务生成依赖结构化模板和槽位规则

### M4：可执行预备态

判断标准：

- 执行器与分发器适配口完成
- 审计与解释模型可落地

## 11. 推荐近期优先级

如果只做最重要的第一波，我建议按下面顺序推进：

1. T4 移除在线自动建 collection
2. T5 建 bootstrap / migration 脚本
3. T6 重构导入链路
4. T7 抽象检索仓储层
5. T8 重构 RagService
6. T10 重构 SQL Node
7. T13 重构 Task Node
8. T18 重构健康检查

## 12. 结论

DataPilot 下一阶段最应该做的，不是继续补更多 Prompt 技巧，而是先把“数据资产结构、检索结构、服务边界、扩展接口”彻底理顺。

只要第二阶段和第三阶段做扎实，这个项目就会从“有想法的 demo”进入“专业 Agent 后端”的状态。

在此基础上，再接 SQL 执行器、任务调度平台和更多数据知识资产，风险会小很多，扩展也会顺很多。
